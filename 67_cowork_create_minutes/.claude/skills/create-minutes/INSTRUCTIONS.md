# create-minutes 詳細手順

NEVER 途中で停止しない。ステップ1〜7をすべて完了するまで実行を継続すること。
「No response requested」や中間報告で終了するのは禁止。最終成果物（議事録ファイル）を書き出して完了報告するまで止まらない。

すべての操作はプロジェクトルートから実行すること。

```text
OUTPUT_PATH = 05_Meetings/{YYYY}/{MM}/{DD}/{filename}.md
```

`{filename}` はイベントの summary をそのまま使用する。

---

## ステップ1: 引数解析と日付決定

```text
引数1: TARGET_DATE (YYYY-MM-DD形式、省略時は実行日)
```

日付が省略された場合:

```bash
TARGET_DATE=$(TZ=Asia/Tokyo date +%Y-%m-%d)
```

## ステップ1.5: deferred tools のスキーマ取得（CRITICAL - NEVER SKIP）

このステップを省略・変形すると議事録生成が 100% 失敗する。NEVER キーワード検索で代替しない。

`google_drive_fetch` と `gcal_list_events` は deferred tool であり、呼び出す前に MUST ToolSearch でスキーマを取得する。

MUST 以下の ToolSearch を「そのままコピー」して実行する。独自のクエリに書き換えない:

```json
[
  {
    "query": "select:mcp__<google_drive_server_id>__google_drive_fetch",
    "max_results": 1
  },
  {
    "query": "select:mcp__<google_calendar_server_id>__gcal_list_events",
    "max_results": 1
  }
]
```

ツール名の UUID 部分は MCP サーバーの設定により変わる可能性がある。
`select:` で取得できない場合は、`available-deferred-tools` リストから `google_drive_fetch` / `gcal_list_events` を含むツール名を探し、その完全名で再度 `select:` を実行する。

### 取得結果の判定と遅延リトライ

MCP サーバーの遅延初期化により、初回の ToolSearch で google_drive_fetch が取得できないことがある（gcal_list_events とは別の MCP サーバーが提供するため）。

- gcal_list_events が取得できていれば、google_drive_fetch が未取得でもステップ2-3に進んでよい
- google_drive_fetch は MUST ステップ4の直前で再取得する（下記「ステップ3.5」参照）
- gcal_list_events も取得できない場合はユーザーに MCP サーバーの接続確認を依頼して終了する

NEVER WebFetch や gcloud CLI にフォールバックしない（認証エラーになる）。

---

## ステップ3.5: google_drive_fetch の遅延取得（ステップ1.5で未取得の場合のみ）

ステップ1.5 で google_drive_fetch のスキーマが取得できなかった場合、ここでリトライする。
ステップ1.5 で既に取得済みの場合はこのステップをスキップする。

最大3回リトライする。各リトライの間に20秒の sleep を入れる（Google Drive MCP サーバーは OAuth token リフレッシュ等で初期化に数十秒かかることがある）:

```text
リトライ1: ToolSearch (select:...__google_drive_fetch) → 成功なら続行
  ↓ 失敗
sleep 20
リトライ2: ToolSearch (select:...__google_drive_fetch) → 成功なら続行
  ↓ 失敗
sleep 20
リトライ3: ToolSearch (select:...__google_drive_fetch) → 成功なら続行
  ↓ 失敗
ユーザーに Google Drive MCP サーバーの接続確認を依頼して終了する
```

## ステップ2: Google Calendar からイベント取得

`gcal_list_events` で TARGET_DATE のイベントを取得する。

パラメータ:

- `timeMin`: `{TARGET_DATE}T00:00:00`
- `timeMax`: `{TARGET_DATE}T23:59:59`
- `timeZone`: `Asia/Tokyo`
- `condenseEventDetails`: `false`（添付ファイル・参加者情報が必要）

## ステップ3: 文字起こし付きイベントの特定

取得したイベントから、以下の条件で議事録対象を絞り込む:

1. `attachments` に `mimeType: "application/vnd.google-apps.document"` かつ `title` に「Gemini」を含むドキュメントがあるイベント
2. 既に議事録が生成済みのイベントはスキップする（後述の重複チェック参照）

### 重複チェック

OUTPUT_PATH のディレクトリ配下に同一イベントの議事録が既に存在する場合はスキップする。
Glob で `05_Meetings/{YYYY}/{MM}/{DD}/*.md`（OUTPUT_PATH のディレクトリ部分）を検索し、ファイル名がイベントの summary と一致するものがあれば生成済みと判定する。
全候補がスキップ対象の場合は「すべて生成済みです」と報告して終了する。

### 候補が複数ある場合（サブエージェント並列処理）

候補イベントが複数ある場合は、Agent ツールでサブエージェントを並列起動し、各イベントの議事録を同時に生成する。
単一の場合はサブエージェントを使わず、そのままステップ4以降を実行する。

並列処理の手順:

1. 各イベントの Gemini メモを `google_drive_fetch` で事前に取得する（1イベントずつ順番に取得）
2. テンプレートを Read ツールで読み込む
3. Agent ツールで全イベントを同時に起動する（1つの応答内に複数の Agent 呼び出し）

各サブエージェントへの prompt に含める情報:

- イベント名（summary）、日付、参加者リスト（attendees）
- Gemini メモの全文テキスト（まとめ・詳細・文字起こし）
- テンプレートの全文
- 出力先パス（冒頭の OUTPUT_PATH に従う）
- ステップ5（議事録生成ルール）とステップ6（ファイル書き出しルール）の全内容をそのまま含める

サブエージェントの設定:

```text
subagent_type: general-purpose
        mode: bypassPermissions
description: "{会議名}の議事録生成"
```

サブエージェント完了後、メインエージェントが全ファイルパスをまとめて報告する。

## ステップ4: Google Drive からGeminiメモを取得

対象イベントの添付ファイルから Gemini メモの `fileId` を抽出し、`google_drive_fetch` で取得する。
複数イベントがある場合でも1件ずつ順番に取得する（同時取得はタイムアウトの原因になる）。

```text
document_ids: ["{fileId}"]
```

取得したドキュメントの `text` フィールドに「まとめ」「詳細」「文字起こし」が含まれる。
議事録生成には「文字起こし」セクションを主に参照する。

---

## ステップ4.5: 過去コンテキストの収集（任意）

議事録生成の精度を上げるため、直近の議事録から未完了 Todo と決定事項を収集できる。

OUTPUT_PATH のルートディレクトリ（`05_Meetings/`）配下から直近3件の議事録を Read し、以下を抽出する:

- 未完了 Todo: `- [ ]` で始まる行（担当者・内容・記録日付を保持）
- 主な決定事項: `## 決定事項` セクションの内容（直近1件のみ）

ファイルが存在しない場合（初回など）はスキップして次のステップへ進む。

---

## ステップ5: 議事録を生成

テンプレートを Read ツールで読み込む:

```text
${CLAUDE_SKILL_DIR}/references/template_default.md
```

`${CLAUDE_SKILL_DIR}/references/feedback.md` が存在する場合は Read で読み込み、過去のフィードバックを議事録生成に反映する。

テンプレートの構造に従い、文字起こし内容を元に議事録を生成する。

まず Gemini メモの「まとめ」「詳細」セクションで会議全体の流れ・論点を把握し、その文脈を踏まえて「文字起こし」セクションの発言内容を正確に解釈すること。文字起こしは断片的・曖昧な発言が多いため、前後の話題の流れや議題の背景から意味を補完して議事録に反映する。

### 生成ルール

- テンプレートの各セクションを埋める
- 参加者はテンプレートに記載されたルールに従って整形する
- Google Calendar のイベント情報（`attendees`）も参加者の補完に使用する
- 決定事項・Todoは明確に抽出する
- テンプレート内の「記載ルール」セクションに従って討議事項を記載する（発言者名・インデント・IT用語英語表記等）
- 討議事項は議論・質疑応答の内容を網羅的に記載する。一方的な説明や報告のみの内容は「その他・内部向けメモ」に記載する

### 参加者の網羅性（MUST）

参加者リストには文字起こしで発言しているすべての人物を MUST 含める。カレンダーの attendees だけでなく、文字起こし内の発言者名を全員確認し、漏れなく記載すること。

### 討議事項の質疑応答はインデントで会話の流れを表現する（MUST）

討議事項では質疑応答・意見交換をフラットに並べず、MUST インデントで会話の流れを表現する。

- 最初の質問・意見: インデントなし `-`
- それに対する回答: スペース4つ `-`
- さらなる追加質問: スペース8つ `-`
- それに対する回答: スペース12つ `-`

### 長い議題は分割する（MUST）

1つの ### 議題が長くなる場合（目安: 10行以上）は、サブトピックごとに ### を分割して可読性を高める。

### 音声認識の誤変換修正（MUST）

Gemini の文字起こしは同音異義語の誤変換が頻発する。文脈から正しい表記に MUST 置き換えること。

よくある誤変換例:

| 誤変換              | 正しい表記 | 判断のヒント           |
| ------------------- | ---------- | ---------------------- |
| タブ賞              | 他部署     | 組織の話題             |
| 朝                  | アサイン   | タスク・人員配置の話題 |
| 順位人 / 準異人     | 準委任     | 契約形態の話題         |
| 受け追い / 受け負い | 請負       | 契約形態の話題         |

上記以外でも不自然な表記があれば、会議全体の文脈から正しい用語を推測して修正する。

## ステップ6: ファイルに書き出す

冒頭の OUTPUT_PATH に従い出力パスを決定する。

ディレクトリが存在しなければ Bash で作成する:

```bash
mkdir -p "$(dirname "${OUTPUT_PATH}")"
```

Write ツールで議事録ファイルを書き出す。

---

## ステップ7: 完了報告とフィードバック収集

書き出したファイルパスを報告する。

報告後、ユーザーにフィードバックを求める:

```text
議事録を確認して、改善点があれば教えてください。
（例: 「Todoの抽出が足りない」「要約が粗い」「用語が違う」等）
フィードバックは次回以降の議事録生成に反映されます。
```

フィードバックがあった場合、`${CLAUDE_SKILL_DIR}/references/feedback.md` に追記する。
ファイルが存在しない場合は新規作成する。

```markdown
## {YYYY-MM-DD} のフィードバック

- {ユーザーからのフィードバック内容}
```

このファイルはステップ5の議事録生成時に自動で読み込まれ、過去のフィードバックが反映される。
