# Integration Test Execution

Reads workflow.json and executes all test steps sequentially.
This skill replaces the Ralph Loop pattern — it acts as the orchestrator
directly, without subprocess invocation or shell scripts.

## Prerequisites

- This skill must run in a `--permission-mode dontAsk` session
- Per-skill hooks in SKILL.md automatically enforce the default-deny
  safety policy during skill execution
- workflow.json must be pre-generated by `/prepare-integration-test`

## AWS Authentication

Authentication is handled automatically by the AWS CLI profile
configuration in `~/.aws/config`. All AWS CLI commands are executed
directly with `aws ... --profile <aws_profile>`.
The CLI resolves credentials according to the profile's settings
(credential_process, SSO, etc.).

No manual authentication steps are needed.
If the credential provider fails, the AWS CLI command will return an
error — stop and report clearly.

## Orchestration Logic

### 0. Load and Validate

1. Read `(project)/workflow.json`
2. Load `aws_profile`, `poll_interval_seconds`, and `config.max_retries`

### 1. Step Traversal

Walk through the `steps` array in order. For each step, check its status:

- `completed` → skip
- `in_progress` + `output` is not null → job already submitted, go to
  completion check (resume from interruption)
- `in_progress` + `output` is null → re-execute from the beginning
  (interrupted before job submission)
- `pending` → execute normally

Process all steps in sequence until all are `completed`.

### 2. Step Execution

For each actionable step:

1. Read the step's `prompt` field for instructions
2. Resolve placeholders in the prompt
   - `{{step-id.output}}`: read the `output` field of the referenced step
   - `{{retry}}`: read the `retry` field of the current step
3. Set `status` to `in_progress` and save workflow.json
4. Execute the instructions using Bash tool with AWS CLI commands
   (include `--profile <aws_profile>` on all AWS commands)
5. Record the result in the `output` field

### 3. Async Job Polling

For asynchronous jobs (Step Functions, Glue, etc.):

1. Run the describe/status-check command once
2. Evaluate the result:
   - Still running (RUNNING, STARTING, PENDING, QUEUED)
     → Record the job ID in `output`, save workflow.json
     → Execute `sleep <poll_interval_seconds>` via Bash tool
     → Check status again (repeat this loop)
   - Succeeded (SUCCEEDED, COMPLETED)
     → Set `status` to `completed`, continue to next step
   - Failed (FAILED, TIMED_OUT, ABORTED)
     → Go to retry logic

For Athena queries, use a fixed 5-second interval (`sleep 5`) instead
of `poll_interval_seconds`. Athena queries typically complete in
seconds, and `poll_interval_seconds` is intended for long-running jobs
like Step Functions.

Unlike the Ralph Loop (which exits after each check and relies on the
shell to re-invoke), this skill polls within the same session using
`sleep` between checks. This is more token-efficient for typical test
suites (5-10 cases).

### 4. Retry Logic

When a step fails:

- If `retry` < `max_retries`
  - Find the paired execute step (replace `-check` with `-execute` in
    the current step's id)
  - Reset both the execute step and current (check) step to `pending`
  - Increment `retry` on both steps
  - Save workflow.json and continue traversal (the reset steps will be
    picked up in the next iteration)
- If `retry` >= `max_retries`
  - Set `status` to `completed` with NG result in `output`
  - Continue to next step

### 5. Evidence Collection (check steps)

During check steps, after confirming job completion:

1. Retrieve execution status (describe-execution, get-job-run, etc.)
2. Retrieve S3 output (list-objects-v2, get-object as needed)
3. Retrieve CloudWatch Logs (filter-log-events)
4. Record the evidence in `(project)/test-outputs/evidences/XX-<test-name>.md`
   - Update the "Actual Value" and "Verdict" columns in the checklist
   - Record the commands and output in the evidence section
5. Update the test case list in `(project)/test-outputs/test-plan.md`
   - If the "Result" column is empty (-) → Update "Result" and "Date"
   - If the "Result" column already has a value → Update
     "Result (retest)" and "Date (retest)"

### 6. Summary Step

The final `summary` step aggregates all test results:

1. Read all evidence files and the test case list
2. Write/update the "Summary" section of test-plan.md
3. Include: total count, OK/NG breakdown, issues detected,
   improvement suggestions

### 7. Completion

When all steps are `completed`, output a final report:

- Total test cases and results (OK/NG counts)
- Path to test-plan.md and evidence directory
- Any issues that need attention

## Interruption and Resume

- workflow.json is the state store — all progress is persisted there
- To resume after interruption: start a new
  `claude --permission-mode dontAsk` session
  and re-run `/run-integration-test (project-path)`
- `completed` steps are skipped
- `in_progress` steps with `output` resume from completion check
- `in_progress` steps without `output` re-execute from the beginning

## Rules

- Execute all steps in a single session (do not exit after each step)
- Use Bash tool's `sleep` for async job waiting (not shell background
  processes)
- Always save workflow.json after every state change
- Include `--profile <aws_profile>` on all AWS CLI commands
- Do not use AskUserQuestion (this skill runs non-interactively)
- Do not invoke subprocesses, shell scripts, or `claude -p`
- Never use `$()` command substitution or backtick substitution in Bash
  commands. `--permission-mode dontAsk` flags `$()` as dangerous and prompts
  for confirmation, which breaks automated execution. Instead, use
  separate Bash tool calls and read the output of the first call to
  pass values to the next call. For example, for Athena queries:
  1. First Bash call: `aws athena start-query-execution ... --output text --query 'QueryExecutionId'`
  2. Read the QueryExecutionId from the output
  3. Second Bash call: `aws athena get-query-results --query-execution-id <id-from-step-1> ...`
